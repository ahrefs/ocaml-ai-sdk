### 3. UIMessage stream composition (`createUIMessageStream` / `createUIMessageStreamResponse`)

**What:** A composable stream builder that lets server code write custom chunks
alongside LLM output in a single SSE response. The upstream TypeScript SDK
provides `createUIMessageStream` (writer with `write()` + `merge()`) and
`createUIMessageStreamResponse` (wraps the stream into an HTTP Response).

**Why it matters:**
- Enables writing custom `Data` parts (e.g. `data-weather`, `data-status`)
  interleaved with `streamText` output in one response.
- Supports data reconciliation â€” writing to the same `id` updates the part
  on the client, enabling loading â†’ success progressive UI patterns.
- `transient` parts (ephemeral, not persisted in message history) for
  notifications and progress indicators.
- `onFinish` callback with `messages`, `responseMessage`, `finishReason`,
  `isContinuation`, `isAborted` â€” useful for persistence and logging.
- `originalMessages` parameter enables persistence mode with automatic
  message ID assignment for the response message.
- Source parts (`source` type) for RAG implementations.

**OCaml design sketch:**

```
(* Writer API *)
module Ui_message_stream_writer : sig
  type t
  val write : t -> Ui_message_chunk.t -> unit
  val merge : t -> Ui_message_chunk.t Lwt_stream.t -> unit Lwt.t
end

(* Stream builder *)
val create_ui_message_stream :
  ?on_error:(exn -> string) ->
  ?on_finish:(finish_options -> unit Lwt.t) ->
  ?original_messages:Yojson.Basic.t list ->
  ?generate_id:(unit -> string) ->
  execute:(writer:Ui_message_stream_writer.t -> unit Lwt.t) ->
  unit ->
  Ui_message_chunk.t Lwt_stream.t

(* HTTP response wrapper â€” thin convenience over make_sse_response *)
val create_ui_message_stream_response :
  ?status:Cohttp.Code.status_code ->
  ?headers:(string * string) list ->
  Ui_message_chunk.t Lwt_stream.t ->
  (Cohttp.Response.t * Cohttp_lwt.Body.t) Lwt.t
```

**Current state:** `server_handler.handle_chat` goes straight from request â†’
`stream_text` â†’ SSE. For custom chunks, users must manually work with
`stream_text` + `to_ui_message_sse_stream` + `make_sse_response`. No
composable writer/merge abstraction exists.

**Key files to modify:**
- New: `lib/ai_core/ui_message_stream_writer.ml` / `.mli`
- Modify: `lib/ai_core/server_handler.ml` â€” `create_ui_message_stream_response`
  can live here or in a new module
- Modify: `lib/ai_core/ui_message_chunk.ml` â€” may need `Data` variant updates
  for `transient` field
- Modify: `lib/ai_core/ai_core.ml` / `.mli` â€” exports

**Upstream references:**
- API: https://ai-sdk.dev/docs/reference/ai-sdk-ui/create-ui-message-stream
- API: https://ai-sdk.dev/docs/reference/ai-sdk-ui/create-ui-message-stream-response
- Usage guide: https://ai-sdk.dev/docs/ai-sdk-ui/streaming-data
- Stream protocol: https://ai-sdk.dev/docs/ai-sdk-ui/stream-protocol

**Depends on:** Nothing (can be built standalone)
**Unlocks:** Transient data parts (#10), tool approval workflow (#4)

### 4. Tool approval workflow

**What:** `tool-approval-request` / `tool-output-denied` chunk types are already
defined in `Ui_message_chunk.t`. Need `needs_approval` field on `Core_tool.t`
and a callback mechanism for the server to pause the tool loop and wait for
approval from the frontend before executing.

**Current state:** Chunk types `Tool_output_denied` and `Tool_input_error`
exist but are unused. `Core_tool.t` has no approval field. The step loop in
`stream_text` executes tools immediately.

**Key files:**
- `lib/ai_core/core_tool.ml` â€” add `?needs_approval:bool`
- `lib/ai_core/stream_text.ml` â€” pause step loop, emit approval request,
  wait for client response
- `lib/ai_core/server_handler.ml` â€” needs a way to feed approval responses
  back into the stream (likely via `Lwt_mvar` or similar)

**Upstream reference:** https://ai-sdk.dev/docs/ai-sdk-core/generating-text#tool-execution

**Depends on:** UIMessage stream composition (#3) is helpful for the
request/response flow but not strictly required.

### 5. Cross-language interop test suite

**What:** Node.js script using `readUIMessageStream` from `ai@6` to consume
our SSE output and verify it parses correctly. Automated CI test.

**Current state:** OCaml-side unit tests exist for chunk serialization. No
end-to-end test that proves a real `useChat()` client can consume our output.

**Approach:**
- Node.js test script in `test/interop/` that starts the OCaml chat server,
  sends requests, pipes responses through `readUIMessageStream`, and asserts
  on the parsed messages.
- Derive test cases from upstream AI SDK test suite where possible.
- Cover: basic text streaming, tool calls, reasoning, structured output,
  error responses, CORS headers.

**Key files:**
- New: `test/interop/test_stream_compat.mjs`
- Reference: `examples/chat_server/` for server setup patterns

**Upstream reference:**
- https://ai-sdk.dev/docs/ai-sdk-ui/reading-ui-message-streams

**Depends on:** Nothing

## Medium Priority

### 6. `smoothStream` text buffering

**What:** Buffer text deltas and emit them at a controlled rate for smoother
UI rendering. Configurable by word boundary, delay, and chunk size.

**Current state:** Text deltas are forwarded as-is from the provider. Some
models emit very small or very large chunks that feel janky in the UI.

**Design:** A stream transformer `Ui_message_chunk.t Lwt_stream.t ->
Ui_message_chunk.t Lwt_stream.t` that buffers `Text_delta` chunks and
re-emits them word-by-word or on a timer.

**Upstream reference:** https://ai-sdk.dev/docs/ai-sdk-core/streaming-text#smooth-text-streaming

**Depends on:** Nothing

### 10. `transient` field on custom `Data` chunks

**What:** The v6 spec supports `transient: true` on `data-*` chunks.
Transient data is sent to the client but not persisted in message history
(useful for progress indicators, notifications).

**Current state:** `Ui_message_chunk.Data` has `data_type`, `id`, `data`
but no `transient` field.

**Key files:**
- `lib/ai_core/ui_message_chunk.ml` â€” add `?transient:bool` to `Data`
