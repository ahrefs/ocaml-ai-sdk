### 2. v6-only request parsing (`convertToModelMessages`)

**Status:** Done

### 3. UIMessage stream composition (`createUIMessageStream` / `createUIMessageStreamResponse`)

**Status:** Done

Implemented `Ui_message_stream_writer` module with composable stream builder API:
- Opaque writer type with `write` (synchronous) and `merge` (non-blocking via

- `create_ui_message_stream` manages lifecycle (Start/Finish/Error chunks),

- `create_ui_message_stream_response` wraps chunk stream as HTTP SSE response

- Minimal `on_finish` with `finish_reason` and `is_aborted` (full message

**Files:** `lib/ai_core/ui_message_stream_writer.ml`, `test/ai_core/test_ui_message_stream_writer.ml`

### 4. Tool approval workflow

**Status:** Done

Implemented stateless tool approval matching upstream AI SDK's `needsApproval` pattern:
- `Core_tool.t` gains `needs_approval : (Yojson.Basic.t -> bool Lwt.t) option` with
  `create` and `create_with_approval` constructors
- Step loops in `generate_text` and `stream_text` partition tool calls: if any tool
  needs approval, the loop stops and emits `Tool_approval_request` chunks
- `server_handler` parses `approval-responded` parts with `approved` field, collects
  pre-approved tool call IDs, and threads them through `~approved_tool_call_ids` so
  the step loop skips re-checking on re-submission
- New `Tool_approval_request` chunk type in `Ui_message_chunk` serialized as
  `tool-approval-request` in SSE stream

### 6. `smoothStream` text buffering

**Status:** Done

Implemented `Smooth_stream` module in `ai_core` matching the upstream AI SDK's
`smoothStream` transform. Buffers `Text_delta` and `Reasoning_delta` chunks and
re-emits them in controlled pieces with configurable inter-chunk delays (default
10ms). Five chunking modes: `Word` (regex `\S+\s+`), `Line` (regex `\n+`),
`Regex` (user-supplied Re2 pattern), `Segmenter` (Unicode UAX#29 word
segmentation via uuseg — recommended for CJK), and `Custom` (user function).
Non-smoothable events flush the buffer and pass through unchanged.

Wired into `stream_text` and `server_handler` via a generic `?transform`
parameter (`Text_stream_part.t Lwt_stream.t -> Text_stream_part.t Lwt_stream.t`).
Both `full_stream` and `text_stream` reflect the transformed output.

19 unit tests (including CJK real-world cases for Segmenter), 2 integration
tests in `test_stream_text`, and a `smooth_streaming` example demonstrating all
five chunking modes.

**Files:** `lib/ai_core/smooth_stream.ml` / `.mli`, `lib/ai_core/stream_text.ml`,
`lib/ai_core/server_handler.ml`, `test/ai_core/test_smooth_stream.ml`,
`examples/smooth_streaming.ml`
**Plan:** `docs/plans/2026-04-08-smooth-stream.md`

### 7. Telemetry / observability

**Status:** Done

Implemented `Telemetry` module in `ai_core` with OpenTelemetry-compatible span
instrumentation via the `trace` library (ocaml-trace 0.12). Configurable
`Telemetry.t` settings control enable/disable, input/output recording privacy,
function ID, custom metadata, and lifecycle integration callbacks.

Span hierarchy matches upstream AI SDK exactly:
- `ai.generateText` → `ai.generateText.doGenerate` → `ai.toolCall`
- `ai.streamText` → `ai.streamText.doStream` → `ai.toolCall`

Attribute keys follow upstream naming (camelCase) and OpenTelemetry `gen_ai.*`
semantic conventions. Input/output attribute selectivity respects
`record_inputs`/`record_outputs` settings with lazy evaluation (zero
serialization cost when disabled). Lwt-aware `with_span_lwt` helper uses
`enter_span`/`exit_span` for correct async span lifecycle.

Integration callbacks: `on_start`, `on_step_finish`, `on_tool_call_start`,
`on_tool_call_finish`, `on_finish`. Global integration registry for
application-wide observability. Errors in callbacks are caught and ignored.

Wired through `generate_text`, `stream_text`, and `server_handler` via
`?telemetry:Telemetry.t` optional parameter. Zero overhead when absent or
disabled.

**Files:** `lib/ai_core/telemetry.ml` / `.mli`, `lib/ai_core/generate_text.ml`,
`lib/ai_core/stream_text.ml`, `lib/ai_core/server_handler.ml`,
`test/ai_core/test_telemetry.ml`, `examples/telemetry_logging.ml`
**Plan:** `docs/plans/2026-04-10-telemetry.md`

### 9. Retry logic with backoff

**Status:** Done

Implemented `Retry` module with exponential backoff matching upstream AI SDK
semantics. Added `is_retryable : bool` field to `Provider_error.t` (replacing
the `"[retryable]"` body prefix hack). `generate_text`, `stream_text`, and
`server_handler` accept `?max_retries` (default 2). Retryable errors
(`Rate_limit_error`, `Overloaded_error`, `Server_error`) are retried with
2s initial delay and 2x backoff. Non-retryable errors and non-Provider
exceptions are re-raised immediately. Exhausted retries wrap in `Retry_error`.

**Files:** `lib/ai_core/retry.ml` / `.mli`, `lib/ai_provider/provider_error.ml` / `.mli`,
`lib/ai_core/generate_text.ml`, `lib/ai_core/stream_text.ml`,
`lib/ai_core/server_handler.ml`

### 19. `providerExecuted` flag support in SSE chunks and message parsing

**Status:** Not started

**What:** The upstream AI SDK has a `providerExecuted` boolean on tool-related SSE
chunks and tool invocation UI parts. It indicates that the **provider (LLM API)
executed the tool server-side** (e.g. MCP tools, provider-native web search),
as opposed to the SDK's tool loop executing it client-side.

This flag affects three areas:

1. **SSE chunk emission** — `tool-input-start`, `tool-input-available`,
   `tool-output-available`, `tool-output-error` chunks all have an optional
   `providerExecuted` boolean field. We currently omit it.

2. **Message parsing (`convert-to-model-messages`)** — When re-parsing
   messages from the frontend, tool results for `providerExecuted` tools go
   into **assistant content** (alongside the `tool-call`), not into a separate
   `Tool` message. Our `parse_tool_result` currently creates `Tool` messages
   for all output-available/output-error states regardless of this flag.
   Upstream explicitly skips `Tool` message creation for provider-executed
   tools (line 280-285: "the tool result is already in the assistant content").

3. **Step loop** — `stream-text.ts` emits `tool-output-available` chunks for
   provider-executed tool results differently (they arrive as `tool-result`
   stream parts from the provider, not from SDK tool execution).

**Current behavior:** We always put tool results in `Tool` messages and never
emit `providerExecuted` on SSE chunks. This is correct for all SDK-executed
tools (which is all we support today), but incorrect for provider-executed
tools.

**Scope:**
- `lib/ai_core/ui_message_chunk.ml` / `.mli` — add optional `provider_executed`
  to tool-related chunk variants and JSON records
- `lib/ai_core/server_handler.ml` — parse `providerExecuted` from tool
  invocation parts; route provider-executed tool results into assistant content
  instead of `Tool` messages
- `lib/ai_core/stream_text_result.ml` — emit `provider_executed` on chunks
  when the provider sets it
- `lib/ai_provider/prompt.ml` — add `provider_executed` to `Tool_call`
  assistant part
- `lib/ai_core/text_stream_part.ml` — add `provider_executed` to `Tool_result`

**Upstream reference:**
- `node_modules/ai/src/ui-message-stream/ui-message-chunks.ts` (chunk schemas)
- `node_modules/ai/src/ui/convert-to-model-messages.ts` (lines 202-230, 251-285)
- `node_modules/ai/src/generate-text/stream-text.ts` (lines 2628-2668)

**Depends on:** #16 (provider metadata, done)

**Blocks:** MCP client (v3 #6), dynamic tools (v3 #3)

### 22. Telemetry parity: missing span attributes and callbacks

**Status:** Not started

**What:** The telemetry implementation (#7) covers the core span hierarchy,
attribute selectivity, and integration callbacks. Several upstream attributes
and one callback are not yet emitted due to missing fields in our types.

**Missing attributes:**
- `ai.response.timestamp` on step spans — `Generate_result.response_info` has
  no timestamp field. Requires adding `timestamp : string option` to
  `response_info` and populating it from the Anthropic provider.
- `ai.response.id`, `ai.response.model` on **stream** step spans — streaming
  does not expose per-step response metadata. Requires threading response
  metadata through `Stream_result` or `consume_provider_stream`.
- `ai.usage.inputTokenDetails.*` (`noCacheTokens`, `cacheReadTokens`,
  `cacheWriteTokens`), `ai.usage.outputTokenDetails.*` (`textTokens`,
  `reasoningTokens`) — `Usage.t` only has `input_tokens`, `output_tokens`,
  `total_tokens`. Anthropic already returns these in provider metadata;
  requires extending `Usage.t` with optional detail fields.

**Missing callback:**
- `on_step_start` — upstream has `onStepStart` fired before each LLM call.
  Requires adding the callback to the `integration` type.

**Documented in:** `lib/ai_core/telemetry.mli` "Upstream parity gaps" section.

**Depends on:** #7

### 20. Retry: respect `retry-after` response headers

**What:** Use `retry-after-ms` and `retry-after` HTTP headers from provider
error responses to determine retry delay instead of pure exponential backoff.
Upstream's `getRetryDelayInMs()` prefers the header value when it is
reasonable (0–60s or less than the exponential delay).

**Current state:** `Retry.with_retries` uses fixed exponential backoff.
`Provider_error.t` does not carry response headers.

**Approach:**
- Add `headers : (string * string) list` to `Api_error` (or to `Provider_error.t`)
- Thread response headers through `anthropic_api.ml` / `openai_api.ml` error paths
- Parse `retry-after-ms` (milliseconds) and `retry-after` (seconds or HTTP-date)
  in `Retry.with_retries`, falling back to exponential delay

**Upstream reference:** `getRetryDelayInMs()` in
`examples/melange_chat/node_modules/ai/src/util/retry-with-exponential-backoff.ts`

**Depends on:** #9

### 21. Retry: abort/cancellation support

**What:** Allow callers to cancel in-flight retry loops. Upstream takes an
`AbortSignal` and stops retrying immediately when the signal fires, recording
the reason `'abort'`. Long `Lwt_unix.sleep` delays currently survive request
aborts.

**Current state:** No cancellation mechanism. A cancelled request's retry
loop continues sleeping and may still call the provider.

**Approach:**
- Add `?switch:Lwt_switch.t` to `Retry.with_retries`
- Check switch state before each retry attempt
- Add `Abort` constructor to `retry_reason`
- Callers (e.g. `server_handler`) can thread a request-lifetime switch

**Depends on:** #9
