# v3 Roadmap — Feature Parity with Vercel AI SDK

> Features remaining after v2 is complete. Organized by category.
> Goal: full feature parity with the upstream TypeScript AI SDK.

---

## Gap Analysis

After v2 is done, the OCaml SDK will have:
- `generate_text` / `stream_text` with tools, multi-step, output API
- UIMessage stream composition (writer + merge)
- Tool approval workflow
- `smoothStream`, `stopWhen`, retry with backoff
- Telemetry, transient data parts
- Provider abstraction, Anthropic provider, middleware interface
- Server handler with CORS
- Cross-language interop tests

What's still missing for parity falls into these categories:

1. **Generation functions** beyond text (embeddings, images, speech, transcription, video)
2. **Agent APIs** (dedicated agent pattern, agent UI streaming)
3. **MCP client** (Model Context Protocol tool integration)
4. **Provider management** (registry, custom providers)
5. **Built-in middleware** (concrete implementations)
6. **Advanced tool features** (activeTools, repairToolCall, prepareStep, dynamicTool)
7. **Message utilities** (convertToModelMessages, pruneMessages, validateUIMessages)
8. **Schema utilities** (jsonSchema, zodSchema equivalents)
9. **Additional providers** (OpenAI, Google, Mistral, etc.)

---

## High Priority

### 1. Embedding model support (`embed` / `embed_many`)

**What:** New model type `Embedding_model.S` in `ai_provider`, plus core
functions `embed` and `embed_many` that call it. Also `cosine_similarity`
utility for comparing vectors.

**Upstream API:**
- `embed(model, value)` → `{ embedding: float array; usage }`
- `embedMany(model, values, ?max_parallel_calls)` → `{ embeddings: float array list; usage }`
- `cosineSimilarity(a, b)` → `float`

**Why high priority:** Embeddings are fundamental to RAG, search, and
classification — the most common AI SDK use cases beyond chat.

**Scope:**
- New: `lib/ai_provider/embedding_model.ml` / `.mli` — model type signature
- New: `lib/ai_core/embed.ml` / `.mli` — `embed` and `embed_many`
- New: `lib/ai_core/cosine_similarity.ml` — vector math utility
- Provider implementation for Anthropic (if/when they ship embeddings) or
  a second provider (e.g. OpenAI)

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/embeddings

---

### 2. Provider registry (`create_provider_registry` / `custom_provider`)

**What:** A registry that maps string IDs like `"anthropic:claude-sonnet-4-5"`
to model instances. Plus `custom_provider` for wrapping providers with
default settings or model aliases.

**Current state:** `Provider.S` exists with `language_model : string -> t`
but there's no multi-provider registry or alias/default mechanism.

**Scope:**
- New: `lib/ai_provider/provider_registry.ml` / `.mli`
- New: `lib/ai_provider/custom_provider.ml` / `.mli`

**Why:** Enables clean multi-provider setups and model aliasing (e.g.
`"fast" → anthropic:claude-haiku-4-5`, `"smart" → anthropic:claude-opus-4-6`).

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/provider-management

---

### 3. Advanced tool features

**What:** Several tool capabilities that the upstream SDK supports but we don't:

| Feature | What it does |
|---------|-------------|
| `activeTools` | Restrict which tools are available per call (subset of `tools`) |
| `repairToolCall` | Callback to fix malformed tool calls before failing |
| `prepareStep` | Pre-process each step before the LLM call (modify tools, system prompt) |
| `dynamicTool` | Tools defined by the model at runtime (LLM picks name + schema) |
| `continueSteps` | Resume a multi-step generation from a previous result |

**Current state:** `Core_tool.t` has `description`, `parameters`, `execute`.
No active-tools filtering, no repair, no dynamic tools.

**Key files:**
- `lib/ai_core/core_tool.ml` — dynamic tool support
- `lib/ai_core/generate_text.ml` / `stream_text.ml` — `active_tools`,
  `repair_tool_call`, `prepare_step`, `continue_steps` parameters

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/generating-text

---

### 4. Message utilities (`toResponseMessages`, `pruneMessages`, `validateUIMessages`)

**What:** Remaining message utilities beyond `convertToModelMessages` (done in v2 #2):

| Function | What it does |
|----------|-------------|
| `toResponseMessages` | Generate result → UIMessage parts for appending to history |
| `pruneMessages` | Remove old messages to fit context window, preserving tool call pairs |
| `validateUIMessages` | Validate message structure before sending |

**Current state:** `convertToModelMessages` is implemented in `server_handler.ml`
(v2 #2). These three utilities are the remaining gap.

**Key files:**
- New: `lib/ai_core/message_pruner.ml` / `.mli`
- Extend: `lib/ai_core/server_handler.ml` — `toResponseMessages`

**Upstream ref:** https://ai-sdk.dev/docs/reference/ai-sdk-ui/convert-to-model-messages

## Medium Priority

### 5. Built-in middleware implementations

**What:** Concrete middleware modules that ship with the SDK:

| Middleware | What it does |
|-----------|-------------|
| `extractReasoningMiddleware` | Separates `<think>` blocks from model output into reasoning field |
| `simulateStreamingMiddleware` | Makes non-streaming models behave like streaming ones |
| `defaultSettingsMiddleware` | Applies default temperature, maxTokens, etc. |
| `addToolInputExamplesMiddleware` | Injects example inputs into tool descriptions |
| `extractJsonMiddleware` | Strips markdown fences from JSON output |

**Current state:** `Middleware.S` module type exists with `wrap_generate` /
`wrap_stream`. `Middleware.apply` works. No concrete implementations ship.

**Key files:**
- New: `lib/ai_core/middleware/` directory with one module per middleware
- Modify: `lib/ai_core/ai_core.ml` — exports

**Upstream ref:** https://ai-sdk.dev/docs/reference/ai-sdk-core/wrap-language-model

---

### 6. MCP client integration

**What:** A Model Context Protocol client that connects to MCP servers and
exposes their tools as `Core_tool.t` values usable with `generate_text` /
`stream_text`.

**Scope:**
- New: `lib/ai_core/mcp_client.ml` / `.mli`
- Transport: HTTP (primary), stdio (local dev)
- `tools()` method returns `(string * Core_tool.t) list` from MCP server
- Resource and prompt listing (secondary)

**Why medium:** MCP is important for ecosystem integration but the OCaml
ecosystem has limited MCP server tooling today. Higher value once more
OCaml MCP servers exist.

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/mcp-tools

---

### 7. Agent APIs

**What:** Dedicated agent pattern beyond `streamText` with `maxSteps`:

| API | What it does |
|-----|-------------|
| `Agent` interface | Contract for agent implementations (system, tools, model selection) |
| `ToolLoopAgent` | Standard agent that loops tools until done |
| `createAgentUIStream` | Stream agent execution to frontend |
| `createAgentUIStreamResponse` | HTTP response wrapper for agent streams |

**Current state:** `stream_text` with `max_steps` covers the basic tool
loop agent pattern. No explicit agent abstraction.

**Design consideration:** In OCaml, this could be a functor or module type
rather than a class hierarchy. The `ToolLoopAgent` is essentially what
`stream_text` already does — the value is in the abstraction for
composing agents with different strategies.

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/agents

---

### 8. Image generation (`generate_image`)

**What:** New model type `Image_model.S` in `ai_provider`, plus core
function `generate_image`.

**Parameters:** `model`, `prompt`, `?size`, `?aspect_ratio`, `?seed`, `?n`,
`?provider_options`

**Returns:** `{ images: image list; warnings }` where `image` has `base64`
and `bytes` accessors.

**Scope:**
- New: `lib/ai_provider/image_model.ml` / `.mli`
- New: `lib/ai_core/generate_image.ml` / `.mli`
- Provider implementation (requires a provider that supports image gen)

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/image-generation

---

## Low Priority

### 9. Speech and transcription

**What:** `generate_speech` (text → audio) and `transcribe` (audio → text).
New model types `Speech_model.S` and `Transcription_model.S`.

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/speech

---

### 10. Video generation (experimental)

**What:** `experimental_generate_video` for text/image → video.

**Upstream ref:** Upstream is also experimental.

---

### 11. Reranking

**What:** `rerank` function that takes a query + documents and returns
relevance-scored results. Requires `Reranking_model.S`.

**Note (ai@6.0.153):** Upstream now accepts plain string model IDs in `rerank()`
(e.g. `'cohere/rerank-v3.5'`), matching the pattern used by `generateText`,
`embed`, etc. Design our `rerank` to accept both `Reranking_model.t` and
string IDs via provider registry from the start.

**Upstream ref:** https://ai-sdk.dev/docs/ai-sdk-core/reranking

---

### 12. Additional providers

**What:** Provider implementations beyond Anthropic:

| Provider | Priority | Notes |
|----------|----------|-------|
| OpenAI | High | Most requested, also covers Azure OpenAI |
| Google (Gemini) | Medium | Growing ecosystem |
| Mistral | Low | Niche but straightforward |
| Bedrock/Vertex | Low | Enterprise, complex auth |

Each provider needs: model implementation, prompt conversion, response
parsing, streaming, tool support, and provider-specific options.

**Current state:** Only `ai_provider_anthropic` exists.

---

### 13. Schema utilities

**What:** Helper functions for defining schemas in OCaml:

- `json_schema` — create JSON Schema from OCaml type descriptions
  (the upstream `jsonSchema()` helper)
- PPX or codegen for deriving JSON Schema from OCaml record types
  (equivalent to `zodSchema()` / `valibotSchema()`)

**Current state:** Schemas are raw `Yojson.Basic.t` values. Users must
hand-write JSON Schema. A PPX that derives schemas from record types would
be a significant DX improvement.

---

### 14. `readUIMessageStream` (client-side)

**What:** A client-side utility that reads an SSE stream and yields
`UIMessage` values. The inverse of `to_ui_message_sse_stream`.

**Why:** Useful for OCaml clients consuming AI SDK streams (e.g. CLI tools,
other OCaml services). The frontend uses the JS version via `useChat()`,
but native OCaml consumers need their own reader.

**Scope:**
- New: `lib/ai_core/read_ui_message_stream.ml` / `.mli`

---

### 15. `useChat` / `useObject` Melange bindings improvements

**What:** The Melange bindings in `bindings/ai-sdk-react/` exist but may
need updates for full v6 feature coverage (parts, tool approvals,
data reconciliation, onData callback for transient parts).

---

### 16. Upstream bump analysis sweep

**What:** Review all `docs/upstream-bump-analysis-*.md` files and implement
minor items that were flagged but deferred as too small to act on at the time
(e.g. optional fields, informational wire-format additions, telemetry attribute
changes). These accumulate across bumps and should be swept periodically so
drift doesn't compound.

**How:** Read each analysis file, check "No Action Required Now" and
"Informational" sections for items that are now actionable (because a related
feature landed or upstream made the field non-optional), and create scoped
work items in the v2 or v3 roadmap as appropriate.

## Feature Parity Summary

After v2 + v3, here's the coverage picture:

| Category | Upstream | After v1 | After v2 | After v3 |
|----------|----------|----------|----------|----------|
| Text generation | `generateText`, `streamText` | Full | Full | Full |
| Structured output | `Output.*` | Full | Full | Full |
| Tool execution | tools, maxSteps, toolChoice | Full | Full + approval, stopWhen | Full + activeTools, repair, dynamic, prepare |
| UIMessage streaming | SSE protocol, chunks | Full | Full + compose/merge | Full |
| Smooth streaming | `smoothStream` | Missing | Full | Full |
| Retry / backoff | Built-in | Missing | Full | Full |
| Telemetry | OpenTelemetry spans | Missing | Full | Full |
| Embeddings | `embed`, `embedMany` | Missing | Missing | Full |
| Image generation | `generateImage` | Missing | Missing | Full |
| Speech / transcription | `generateSpeech`, `transcribe` | Missing | Missing | Full |
| Video generation | `experimental_generateVideo` | Missing | Missing | Full (experimental) |
| Reranking | `rerank` | Missing | Missing | Full |
| Provider registry | `createProviderRegistry` | Missing | Missing | Full |
| Custom provider | `customProvider` | Missing | Missing | Full |
| Middleware (interface) | `wrapLanguageModel` | Interface only | Interface only | Full (5 built-in) |
| MCP client | `createMCPClient` | Missing | Missing | Full |
| Agent APIs | `Agent`, `ToolLoopAgent` | Missing | Missing | Full |
| Message conversion | `convertToModelMessages` | Partial | Full | Full |
| Message pruning | `pruneMessages` | Missing | Missing | Full |
| Providers | 10+ providers | Anthropic only | Anthropic only | +OpenAI, Google |
| Schema helpers | `jsonSchema`, `zodSchema` | Manual JSON | Manual JSON | Helper + PPX |
| Cross-language tests | Node.js interop | Missing | Full | Full |

**Bottom line:** After v2, the SDK covers the core chat/streaming/tools use
case well. v3 fills the remaining ~40% needed for full parity, with
embeddings, provider registry, advanced tools, and multi-modal generation
being the most impactful gaps.
